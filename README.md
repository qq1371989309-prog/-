# 风机仿真替换工具

## 项目简介

本项目用于在离线环境下批量处理 `.map` 文件，通过风机地址映射关系，将**本机（有故障需要仿真的）**的数据替换为**目标风机（被仿的无故障的）**的对应数据。

项目目标不是单次生成某个文件，而是沉淀为一个**可长期维护、可持续修正规则、可直接交付使用**的仓库工程。

## 项目定位

本项目同时包含两部分：

### 1. 可直接运行的成品
用于现场直接执行替换任务。

### 2. 可维护的源码工程
用于后续持续维护以下内容：

- 风机地址映射
- 特殊替换规则
- 报告格式
- 界面样式
- 批处理逻辑

## 核心功能

- 批量扫描 `.map` 文件
- 根据风机地址映射执行替换
- 使用 `.map` 第 5 列作为地址索引
- 仅替换 `.map` 第 1 列数值
- 自动输出替换结果文件
- 自动生成替换报告
- 支持后续维护特殊规则
- 支持离线运行，不依赖互联网

## 术语说明

### 本机（有故障需要仿真的）
指当前需要被修改的风机对象。  
替换完成后，本机的数据行为应尽量接近目标风机。

### 目标风机（被仿的无故障的）
指提供参考数据的风机对象。  
程序会读取目标风机对应地址上的值，并写入本机。

## 当前规则方向

当前版本的全局默认规则为：

- 映射来源：`MODBUSTable.xls` 的 `Device_1_3`
- 跳过：`IEMP`
- 只处理：`.map` 中 `col2=1` 且 `col3=3`
- 索引方式：`(col2, col3, addr)`
- 只替换：第 1 列

这套规则用于避免同地址在不同段里重复出现时发生串改。

## 处理逻辑

项目当前的基础逻辑为：

`本机地址[i] <- 目标风机地址[i]`

处理步骤如下：

1. 读取 `.map` 文件
2. 按 `(col2, col3, addr)` 建立索引
3. 根据风机地址映射逐项对应
4. 将目标风机对应地址的第 1 列值写入本机地址
5. 输出新 `.map`
6. 生成 `.csv` 报告

## 关键文件说明

### `src/app.hta`
项目源码入口。  
用于维护界面、文件处理、替换流程、报告输出等逻辑。

### `release/map_tool.hta`
可直接运行的成品版本。  
适合现场直接使用，不要求阅读源码。

### `data/devices.json`
风机名称列表。  
主要用于界面下拉框和风机选择。

### `data/device_maps.json`
风机地址映射表。  
定义每台风机对应的十进制地址数组，是本项目的核心数据之一。

### `data/rules.json`
规则扩展入口。  
用于维护特殊地址的跳过、重定向等逻辑，避免把大量业务例外直接写死在源码中。

## 报告文件说明

程序执行后，会在 `release/reports/` 中生成替换报告。

报告字段用于解决以下问题：

- 谁仿谁
- 哪个地址对应哪个地址
- 替换前值是什么
- 替换后值是什么

当前报告字段包括：

- 原始文件名
- 替换关系
- 本机风机
- 目标风机
- 本机地址
- 目标地址
- 替换前值
- 替换后值
- 对比

报告的主要用途不是程序消费，而是**人工复核**。

## 使用方式

### 直接使用成品

1. 打开 `release/map_tool.hta`
2. 将原始 `.map` 文件放入 `release/input_maps`
3. 选择本机和目标风机
4. 添加仿真关系
5. 开始批量替换
6. 在 `release/output_maps` 查看结果
7. 在 `release/reports` 查看报告

## 开发与维护

### 修改界面
维护文件：

`src/app.hta`

### 修改风机地址映射
维护文件：

`data/device_maps.json`

### 修改特殊规则
维护文件：

`data/rules.json`

### 重新生成成品
执行：

`build_release.bat`

## 规则维护原则

后续如果发现以下问题：

- 错改
- 漏改
- 特殊地址不能按默认顺序处理
- 个别地址需要从指定地址取值

优先处理方式应为：

1. 先确认样本
2. 再更新 `data/rules.json`
3. 最后才考虑修改核心源码

这样做的目的是让项目保持：

- 规则与逻辑分离
- 易维护
- 易审查
- 易追踪变更
